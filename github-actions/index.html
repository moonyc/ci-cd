GitHub Actions

Key Elements and First Workflow

GitHub Actions is a workflow automation service offered by GitHub:
it can automate all kind of possible repository processes.
You need to understand three fundamental building blocks to efficiently integrate
the service in our pipelines: workflows, jobs and steps.
Workflows, jobs and steps are related to each other and to the code repository like this:

  1. The workflows are attached to the repositories.
    You can add as many workflows as you want to a given repository.
  2. Workflows contain one or more jobs.
  3. Jobs contain one or more Steps
  4. Every job defines a runner, which is the environment, ergo
    the operating system used to execute the steps.
  5. GitHub offer runners for Linux, Mac OS and Windows, but you can configure
    your runners.
  6. If you have multiple jobs, they run in parallel by default
    but you can configure them to run in a sequential order.
  7. You can setup conditional jobs.
  8. Steps define the actual commands that will be executed.
  9. A steps is a shell script or an Action.
  10. Actions are predefined script that perform a certain task
  11. Steps are executed in sequential order.
  12. Steps can be conditional

We build workflows to automate processes, consequentially
we assign a trigger (or event) to each workflow, so that we're able to regulate
their execution's recurrence. An event could require manual activation,
but you could also trigger the workflow whenever a commit is pushed to a certain branch.

A Simple Example:

name: Basic Workflow
// This means that you will have to trigger it manually:
on: workflow_dispatch
jobs:
// Press tab to intend correctly:
  job:
// Search "GitHub Actions runners"
   runs-on: ubuntu-lates
   steps:
    - name: Print Hello
      run: echo "Hello Sasha"
    - name: Print Goodbye
      run: echo "Goodbye Sasha"

Artifacts, Jobs Output, Cache dependencies

  1. Artifacts are Outputs assets generated by jobs
  2. Artifacts are a way to store output assets.
  3. Artifacts are files and folders generated by a job, that we want to use or inspect in other jobs
  4. We can manually download and use artifacts: via github ui or rest apis
  5. We can download and use artifacts in other jobs: via action
  6. Jobs can output simple value that we might wanna use in a subsequent job. A Job Output could be any kind of value that we want to make available across job.
  7. We can cache dependencies
  8. Cached dependencies are used across workflow executions

Important To Note:

  1. Actions and Workflows are not synonimous.
  2. Pull Requests based on forks do NOT trigger a workflow.
  3. Workflows get deleted if Jobs fail
  4. A job fails if at least one step fails
  5. Workflows can be cancelled manually
  6. Workflows can be skipped manually
